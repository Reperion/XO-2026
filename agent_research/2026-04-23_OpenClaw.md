## Key Patterns Observed

OpenClaw uses a hierarchical AGENTS.md system:
- **Root AGENTS.md**: High-level rules, repo map, architecture principles
- **Scoped AGENTS.md**: Subdirectory-specific rules (extensions/AGENTS.md, src/channels/AGENTS.md, etc.)

### 2. Repository Map

OpenClaw clearly documents their structure:

### 3. Architecture Rules

OpenClaw enforces:
- Core must stay extension-agnostic
- Extensions use Plugin SDK contracts, not core internals
- Manifest-first control plane
- Backwards-compatible seams

**Lesson for XO**: As an agent running on Hermes:
- I should NOT modify core Hermes code (already in my rules)
- My skills should be self-contained in `~/.hermes/skills/`
- My workspace state lives in `/home/lucid/xo/` (files, not memory)
- I should use Hermes tools, not bypass them

### 4. Standard Commands

OpenClaw defines:
- Install: `pnpm install`
- Build: `pnpm build`
- Test: `pnpm test`
- Typecheck: `pnpm tsgo`
- Lint: `pnpm lint`

**Lesson for XO**: I need standard commands:
- Sync to GitHub: `/home/lucid/xo/sync.sh`
- Skill create: `hermes skills create` or `skill_manage(action='create')`
- Project board update: update ACTIVE.md, PROJECTS.md
- Self-review: use `session_search`, write to `nightly_reports/`

### 5. GitHub API Usage

OpenClaw uses `gh` CLI for CI polling with specific field requests:
```bash
gh api repos/<owner>/<repo>/actions/runs/<id> --jq '{status,conclusion,head_sha}'
```

**Lesson for XO**: I should use `gh` CLI for GitHub operations, not raw API calls. Already in my skills.
